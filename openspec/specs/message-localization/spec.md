# Message Localization

## Purpose

All agent-generated messages and localized room UI content SHALL respect each participant's language preference (`en` or `uk`).

## Requirements

### Requirement: Per-participant locale preference

Each side participant SHALL have a stored language preference of `en` (English) or `uk` (Ukrainian). The preference SHALL be set from the participant's locale choice in the portal (existing LocaleSwitcher) and persisted on the user record for server-side pipeline use.

#### Scenario: Locale persisted on switch

- **WHEN** a participant changes language via the LocaleSwitcher
- **THEN** `preferred_locale` on their user record is updated to `en` or `uk`
- **AND** subsequent AI messages directed at that participant use the updated preference

#### Scenario: Default locale on first visit

- **WHEN** a participant has no stored `preferred_locale`
- **THEN** the system uses browser-detected locale (`en` or `uk`) and persists it on first room or portal interaction

### Requirement: Private thread AI messages in recipient locale

All agent messages in a private thread SHALL be generated in the recipient participant's `preferred_locale`.

#### Scenario: Agent 1 jurisdiction question language

- **WHEN** Agent 1 sends a jurisdiction question to Side 1 whose `preferred_locale` is `uk`
- **THEN** the private-thread message content is in Ukrainian

#### Scenario: Agent 4 clarification language

- **WHEN** Agent 4 asks a clarifying question to Side 2 whose `preferred_locale` is `en`
- **THEN** the private-thread message content is in English

### Requirement: Shared room AI messages per viewer locale

Agent messages published to the shared room (resolution options, post-resolution AI replies, system status messages generated by agents) SHALL be displayed to each participant in their own `preferred_locale`. When room participants have different locale preferences, the system SHALL store localized content for each required locale and resolve the correct version at display time.

#### Scenario: Same locale in room

- **WHEN** all sides in a room have `preferred_locale = uk`
- **THEN** shared-room agent messages are generated and stored in Ukrainian
- **AND** all participants see Ukrainian content

#### Scenario: Mixed locales in room

- **WHEN** Side 1 prefers `uk` and Side 2 prefers `en`
- **THEN** shared-room agent messages (e.g. three resolution options) are stored with both `en` and `uk` content
- **AND** Side 1 sees the Ukrainian version
- **AND** Side 2 sees the English version

### Requirement: Agent locale instruction

Every agent invocation for participant-facing output SHALL include the target locale (`en` or `uk`) in the prompt context so the model responds in the correct language.

#### Scenario: Pipeline passes locale to agent

- **WHEN** the orchestrator runs an agent that produces a private-thread message
- **THEN** the agent call includes `targetLocale` set to the recipient's `preferred_locale`
- **AND** the generated text is in that language

### Requirement: Room UI chrome localized

Participant room UI labels, buttons, placeholders, and system status strings (non-agent) SHALL use the existing portal i18n copy (`en` / `uk`) based on the viewing participant's locale preference.

#### Scenario: Room UI follows locale

- **WHEN** a participant with `preferred_locale = uk` opens the room
- **THEN** tab labels, form labels, and status banners are shown in Ukrainian
- **AND** agent message content follows the locale rules above

### Requirement: Participant-authored messages unchanged

Participant-submitted messages (situation descriptions, shared-room posts, private-thread replies) SHALL be stored as written. The system SHALL NOT auto-translate participant-authored content.

#### Scenario: Participant writes in preferred language

- **WHEN** a participant submits a situation description in Ukrainian
- **THEN** the stored text remains in Ukrainian regardless of the other side's locale preference
